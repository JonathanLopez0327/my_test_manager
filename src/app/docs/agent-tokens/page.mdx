# Agent tokens & RBAC

An **agent token** is a long-lived bearer credential that represents a specific user + organization. Every API call made with the token runs as if that user made it — same permissions, same audit trail.

## Creating a token

From **Settings → API Tokens** (organization owners or admins only):

1. Click **New token**
2. Name it (e.g. `ci-runner-prod`)
3. Pick the user it should impersonate (defaults to you)
4. Copy the token shown **once** — it cannot be retrieved later

Tokens have format `agt_<random>` and are signed/verified server-side.

## Using a token

Send the token as a `Bearer` token in the `Authorization` header:

```http
Authorization: Bearer agt_01HXX...
```

That&apos;s it. There is no separate "agent API" — the same routes documented in the [API reference](/docs/api) accept either session cookies (interactive use) or agent tokens (programmatic use).

## Permissions

Agent tokens are **strictly scoped**:

- Limited to one organization (the org active when the token was minted)
- Cannot create new tokens
- Cannot change user roles
- Cannot delete the user, the organization, or other tokens

Beyond those server-enforced constraints, the agent has the same permissions as the user it represents. Need narrower access? Create a dedicated user with a `viewer`/`editor` role and mint the token for that user.

## Rotation

When you suspect a token has leaked:

1. Revoke it in **Settings → API Tokens**
2. Mint a new one
3. Update your CI/integration secrets

<Callout tone="warning" title="No silent fallback">
  Revoked tokens return HTTP 401 immediately. The integration won&apos;t auto-recover — you must rotate the secret.
</Callout>

## Audit log

Every API call made with an agent token shows up in the organization audit log with the token name and the user it represents. Use this to investigate "who actually did this".
